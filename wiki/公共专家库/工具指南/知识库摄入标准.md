# 知识库摄入标准说明

> 本文档定义 OPCAgent Vault 的所有知识摄入方式、路径选择和标准流程。

---

## 一、摄入路径全景

OPCAgent 有两类知识来源，对应两条完全不同的摄入路径：

```
┌─────────────────────────────────────────────────────────────┐
│                      知识摄入全景                             │
│                                                              │
│  【路径A】公共专家知识（已有MD文件）                            │
│   核心团队 → Git 仓库 → 同步到各主机 → 直接落盘 wiki/         │
│                                                              │
│  【路径B】运行动态知识（Agent 生成）                           │
│   Web 页面 → Claude CLI → MCP Server → wiki/                │
│   产品信息 / 产品报告 / 爆品分析 / 数据归因 / 任务+脚本        │
└─────────────────────────────────────────────────────────────┘
```

| 维度 | 路径A：公共专家知识 | 路径B：运行动态知识 |
|------|-------------------|-------------------|
| **来源** | 核心团队编写、外部文章剪藏 | Web Agent 页面 AI 生成 |
| **文件形态** | 已有的 `.md` 文件 | Agent 实时生成的 Markdown |
| **写入方式** | Git 同步 / 手动放置 | MCP Server (`wiki_create_page`) |
| **同步策略** | 所有主机保持一致 | 每台主机独立 |
| **触发者** | 人类 | Agent（自动） |
| **目录** | `wiki/公共专家库/` | `wiki/产品库/` `wiki/爆品分析/` `wiki/任务中心/` `wiki/数据归因/` |

---

## 二、路径A：公共专家知识的摄入

### 2.1 适用场景

| 知识类型 | 示例 | 目标目录 |
|---------|------|---------|
| 平台规则 | TikTok 广告政策、Shopee 禁售清单 | `wiki/公共专家库/平台规则/` |
| 投放策略 | 冷启动 SOP、出价方法论、扩量策略 | `wiki/公共专家库/投放策略/` |
| 内容方法论 | AIDA 公式、分镜原则、文案框架 | `wiki/公共专家库/内容方法论/` |
| 工具指南 | OPCAgent 操作手册、CLI 配置指南 | `wiki/公共专家库/工具指南/` |
| 术语定义 | 电商/投放/量化领域术语 | `wiki/公共专家库/术语表.md` |

### 2.2 摄入方式

公共专家知识通过 **Git 版本管理** 分发到各主机，不走 MCP：

```
核心团队维护 Git 仓库
    │
    │  git push
    ▼
Git 远程仓库 (wiki/公共专家库/)
    │
    │  各主机 git pull
    ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│  主机 A   │  │  主机 B   │  │  主机 C   │
│ wiki/公共 │  │ wiki/公共 │  │ wiki/公共 │
│ 专家库/   │  │ 专家库/   │  │ 专家库/   │
└──────────┘  └──────────┘  └──────────┘
```

### 2.3 新增一篇公共专家知识的标准流程

**第 1 步：创建 MD 文件**

在 `wiki/公共专家库/` 对应子目录下创建 `.md` 文件，必须包含 frontmatter：

```yaml
---
title: "TikTok 日本市场广告政策 v2026"
type: "rule"
tags: ["TikTok", "日本", "广告政策", "平台规则"]
created: 2026-06-14
modified: 2026-06-14
status: "active"
version: "1.0"
sources: []
---
```

**第 2 步：文件命名**

| 内容类型 | 命名规则 | 示例 |
|---------|---------|------|
| 平台规则 | `{平台}-{主题}-{版本}.md` | `TikTok-日本广告政策-v2026.md` |
| 投放策略 | `{策略名}-方法论.md` | `冷启动-三阶段方法论.md` |
| 内容方法论 | `{公式名}.md` 或 `{概念名}.md` | `AIDA公式.md` |
| 工具指南 | `{工具名}-使用指南.md` | `Claude-Code-使用指南.md` |

**第 3 步：写内容时遵守**

- 使用 `[[wikilinks]]` 引用相关概念页面
- 引用 raw/ 中的原始资料使用 `[[raw/外部知识/文件名]]`
- 正文遵守 `app/standard/knowledge-md-format-standard.md`（标题、段落、列表、代码块、表格）

**第 4 步：维护术语表**

如果文章中引入了新术语，同步更新 `wiki/公共专家库/术语表.md`：

```markdown
| 术语 | 英文 | 定义 | 出处 |
|------|------|------|------|
| 冷启动 | Cold Start | 新广告账户/新产品在无历史数据时的初始投放阶段 | [[冷启动-三阶段方法论]] |
```

**第 5 步：Git 提交 + 各主机同步**

```bash
git add wiki/公共专家库/
git commit -m "docs: 新增 TikTok 日本广告政策 v2026"
git push

# 各主机执行
git pull
```

### 2.4 批量导入外部知识（网页剪藏等）

如果是从网页剪藏的 `.md` 文件（如 Obsidian Web Clipper 产出）：

1. 放入 `raw/外部知识/`（保持 Clipper 自动命名）
2. Agent 执行 `wiki-linter` 时会检测到，或手动触发 `product-ingestor`
3. Agent 读取 raw/ 文件 → 判断归属 → 创建对应 wiki 页面
4. 原始 raw/ 文件保留不删

## 三、路径B：运行动态知识的摄入（Agent 自动）

### 3.1 适用场景

| 知识类型 | 生成者 | 目标目录 | 触发时机 |
|---------|--------|---------|---------|
| 产品档案 | 产品管理 Agent (8766) | `wiki/产品库/产品档案/` | 人类在 Web 页面录入产品信息 |
| 产品报告 | 报告管理 Agent (8766) | `wiki/产品库/产品报告/` | 产品档案创建/更新后 |
| 爆品分析 | 爆品分析 Agent (8768) | `wiki/爆品分析/分析报告/` | raw/爆品素材/ 有新文件 |
| 任务记录 | 任务控制 Agent (8771) | `wiki/任务中心/任务记录/` | 人类在 Web 页面创建任务 |
| 脚本文件 | 脚本管理 Agent (8767) | `wiki/任务中心/脚本库/` | 任务启动后 AI 生成 |
| 归因报告 | 归因分析 Agent (8769) | `wiki/数据归因/归因报告/` | 投放数据可用时 |
| 数据洞察 | 归因分析 Agent (8769) | `wiki/数据归因/数据洞察/` | 从归因数据中提炼 |

### 3.2 摄入流程（以产品档案为例）

```
人类在 Web 页面填写产品信息
    │
    ▼
Web 后端生成 JSON（按 parameter-io-standard.md 格式）
    │
    ▼
Web 后端把 JSON 写入 raw/产品原始资料/{productCode}.json
    │
    ▼
Web 后端触发 Claude CLI 子进程：
  claude -p --bare @System-Prompt-模板库.md "为 productCode=XXX-001 创建产品档案"
    │
    ▼
Claude CLI 子进程：
  1. 读取 @CLAUDE.md → 了解规则
  2. 读取 raw/产品原始资料/{productCode}.json → 获取产品数据
  3. 调用 MCP wiki_create_page(type="product", title="产品中文名", content="...")
    │
    ▼
MCP Server：
  1. 创建 wiki/产品库/产品档案/{产品中文名}.md
  2. 自动生成 frontmatter
  3. 更新 wiki/index.md 统计
  4. 追加 log.md
    │
    ▼
完成 → Web 页面收到成功信号 → 刷新产品列表
```

### 3.3 MCP 写入工具速查

Agent 通过 MCP 写入知识库时，使用以下工具：

| 工具 | 用途 | 示例 |
|------|------|------|
| `wiki_create_page` | 新建 wiki 页面 | 创建产品档案、报告、分析、脚本 |
| `wiki_update_page` | 更新已有页面 | 更新产品档案的 report_versions、任务状态 |
| `wiki_ingest` | 从 raw/ 摄入 | raw/ 新文件自动编译到 wiki/ |
| `wiki_delete_page` | 安全删除 | 删除过时报告（检查入链） |

### 3.4 Agent 写入时必须遵守的规则

| 规则 | 说明 |
|------|------|
| **通过 MCP 写入** | 不允许直接 Write/Edit wiki/ 下的文件 |
| **Frontmatter 必填** | title / type / tags / created / status / sources |
| **命名规范** | 遵守 CLAUDE.md 第三章的命名约定 |
| **双向链接** | 写入后建立与相关页面的 `[[wikilinks]]` |
| **更新 index** | MCP 自动更新，不需要 Agent 手动操作 |
| **版本化** | 产品报告用递增版本号，旧报告标记 inactive |
| **禁止覆盖** | 创建前检查是否已存在同名文件 |

## 四、raw/ 层的角色

`raw/` 是两条路径的交叉点：

```
路径B的入口：
  人类在 Web 页面填数据 → JSON 写入 raw/产品原始资料/ → Agent 读取 → MCP 写入 wiki/

路径A的补充：
  网页剪藏 .md → 放入 raw/外部知识/ → Agent 判断归属 → MCP 写入 wiki/
  爆款视频素材 → 放入 raw/爆品素材/ → Agent 分析 → MCP 写入 wiki/爆品分析/
```

| raw/ 目录 | 写入者 | 读取者 | 用途 |
|-----------|--------|--------|------|
| `raw/产品原始资料/` | Web 后端 | Agent (product-ingestor) | 产品信息 JSON + 图片 |
| `raw/竞品参考/` | 人类 | Agent (综合分析) | 竞品分析素材 |
| `raw/爆品素材/` | 人类 | Agent (hot-product-analyzer) | 爆款视频描述和截图 |
| `raw/外部知识/` | 人类 (Web Clipper) | Agent (wiki-linter) | 网页剪藏和外部文档 |

> ⚠️ **raw/ 是只读的**。Agent 读取后不修改、不删除 raw/ 中的文件。处理完成的 raw/ 文件由人类手动清理。

## 五、摄入后的验证

每次摄入完成后，验证以下内容：

| 检查项 | 方法 |
|--------|------|
| 文件是否在正确位置 | `find wiki/ -name "*.md" | sort` |
| Frontmatter 是否完整 | 打开文件，确认有 title / type / tags / created / status |
| index.md 是否更新 | 打开 `wiki/index.md`，确认统计表已更新 |
| wikilinks 是否有效 | Agent 执行 `wiki-linter` |
| 内容格式是否正确 | 抽查文件，确认遵守 `knowledge-md-format-standard.md` |

## 六、常见问题

### Q: 公共专家知识可以用 MCP 写入吗？

可以但不推荐。公共专家库需要 Git 同步到所有主机，MCP 写入只会影响本机。正确做法是通过 Git PR → review → merge → pull 到各主机。

### Q: 产品报告能手动编辑吗？

不推荐。产品报告由 AI 生成，手动编辑会导致下次 AI 读取时认知不一致。如需修正，应更新 raw/ 中的产品原始资料，然后重新生成报告。

### Q: raw/ 中的文件处理完能删吗？

能。Agent 编译到 wiki/ 后，raw/ 中的原始文件可以手动清理。但建议保留一段时间作为溯源依据。

### Q: 如果 MCP Server 挂了，Agent 怎么写入？

降级方案：Agent 直接将 MD 文件写入 `wiki/`，然后手动触发 `wiki_rebuild_index` 重建索引。但这是应急方案，正常情况下必须走 MCP。
